# 싱송(SingSong) — 코인노래방 세션 플래너

> **한 줄 정의**: 노래방 가기 전에 부를 곡을 담으면 **예상 시간·금액·1인당 요금**을 티켓 한 장으로 계산해 주고, 그 티켓을 카톡으로 보내 **"노래방 가자, 모여!"** 를 만드는 앱.
>
> **주 타깃**: 코인노래방 (메인) + 룸노래방 (확장) · 2030 혼코노/소그룹
> **플랫폼**: 웹(PWA) 우선 → 안드로이드 → iOS
> **인증**: **익명 우선 + 지연 이메일 가입** (첫 진입 무가입 → 결제·다기기 동기화 시에만 passwordless 이메일)
> **문서 버전**: v1.3 · 2026-07-21 · 작성: 앱 개발 총괄
> *(v1.1: 코드리뷰 반영 — 묶음요금 최저가 공식, 웹결제 어뷰징/CS 대응, tags SQLite 호환)*
> *(v1.2: 신원 전략을 '익명 우선 + 지연 이메일 가입(Passwordless)' progressive 모델로 전환)*
> *(v1.3: 플리 공유(링크+스샷 이미지) + 받으면 내 플리로 저장(fork) 기능 추가, 공유 모델 fork/live 구분)*
> *(v1.4: 발견(Discover) 탭 + 셀럽/큐레이션 플리 기능 추가, 팬덤 유입 훅·퍼블리시티권 주의)*

## 0. 문서 목적

이 문서는 지금까지의 모든 기획·설계 의사결정과 그 근거를 한곳에 모은 **단일 기준 문서(Source of Truth)** 다. 시장·경쟁·기술·비용·디자인·로드맵을 포함하며, 개발 착수 시 이 문서를 기준으로 태스크를 쪼갠다.

---

## 1. TL;DR (핵심 결론)

- **포지셔닝 공백이 확실하다.** '부르는 앱'(TJ·금영·Smule)과 '단순 번호검색 앱'은 포화지만, **"가기 전 계획 + 시간/금액 계산 + 친구 모으기"** 를 묶은 앱은 사실상 없다.
- **핵심 가치는 '관리'가 아니라 '계획+정산+모으기'.** "애창곡 관리 유틸"로는 무료 앱·메모장을 못 이긴다. **계산기 + 티켓 공유**가 대체 불가한 차별점이다.
- **개발 난이도: 중.** AI 보조 1인으로 가능. 진짜 난제는 코드가 아니라 ① 곡번호 DB 확보(법적 회색지대) ② (원래) 스토어 심사였는데, **로그인을 없애면서 심사 부담의 상당 부분이 사라진다.**
- **인프라 원가 극도로 낮음** (텍스트 데이터 중심). 서버비 손익분기는 MAU 1만 이전.
- **최대 성장 엔진 = 공유 링크 자체.** 로그인 장벽이 없어 바이럴 계수가 극대화된다.

## 2. 문제 정의 & 타깃

### 2-1. 해결하는 통증
| 통증 | 현재 사용자의 대처(대체재) | 한계 |
|---|---|---|
| "노래방에서 뭐 부르지" 기억 안 남 | 멜론 플리 + 폰 메모장 | 노래방 번호·키가 없어 현장 재검색 |
| 이 곡들 다 부르려면 시간·돈 얼마? | 머릿속 어림짐작 | 시간 모자라거나 남거나, 정산 애매 |
| 친구들끼리 뭐 부를지 / 언제 갈지 | 카톡으로 중구난방 | 합쳐서 순서·시간 정하는 도구 없음 |
| 코인노래방 곡당/시간 요금 계산 | 안 함 | 예산 감이 없음 |

### 2-2. 타깃 페르소나
- **주**: 코인노래방 자주 가는 20대(대학생·사회초년생), 혼코노 + 2~4인 소그룹.
- **부**: 회식·모임으로 룸노래방 잡는 3040 (룸 시간제 모드).

### 2-3. 핵심 인사이트
> 노래방에서 "아 이거!" 하고 곡이 떠오르는 건 **가사를 읽어서가 아니라 평소에 듣던 곡이라서**다. → 그래서 "가사"보다 **플레이리스트 임포트**가 선곡 문제의 근본 해법이다.

## 5. 차별화 / 경쟁 승리 포인트

우선순위(임팩트 × 방어력) 순:

1. **세션 계산기 (시간·금액·1인당)** ★핵심 — 리모컨·음원앱·메모장 전부 못 하는 "가기 전 계획". 정방향(곡→시간·돈) + **역방향(예산→곡수)** 둘 다.
2. **티켓형 공유카드 + 플리 저장(fork) = 바이럴 루프** ★ — 공유되는 게 정리된 리스트가 아니라 "놀 계획". 카톡/인스타 캡처에 강하고, **받은 사람이 "내 플리로 저장"하며 루프가 이어짐**(공유→저장→재공유).
3. **스트리밍 플리 → TJ/KY 번호 자동 매핑** ★온보딩 훅 — 기존 습관을 그대로 흡수. "기억 안 남" 문제의 근본 해법.
4. **TJ/KY 통합 + 개인 노트(키·최고음·상황태그)** — 공식 앱은 자사 전용.
5. **BM 차별화** — 무료 코어 + 라이트 구독(월 1~2천원). 경쟁 유틸의 광고 피로 회피.
6. **셀럽/팬덤 플리 발견** ★유입 훅 — 아이돌·가수의 노래방 애창곡 플리(트위터에서 이미 바이럴)를 **번호 포함·저장 가능**한 형태로. fork 바이럴 + 케이팝 팬덤이라는 최상위 오가닉 유입 채널. (§7-5)

**핵심 프레이밍**: "애창곡 관리"(약함) → **"노래방 세션 플래너"**(강함). "음원앱과 뭐가 다르냐" 질문이 사라진다(음원앱은 계획·정산·모으기를 안 하니까).

## 6. 제품 컨셉 & 핵심 흐름

```
① 담기      → ② 계산       → ③ 모으기 / 공유
부를 곡 검색   시간·금액·1인당   티켓·플리 공유(링크+이미지)
+ 플리 임포트   코인/룸 모드      "노래방 가자!" 모으기
+ 태그/노트    예산→곡수 역계산   받으면 → 내 플리로 저장(fork)
```

세 화면이 전부. 검색·관리는 재료, 주인공은 **계산과 티켓**.

- 실시간 그룹 큐(방코드·QR, 동시편집·순서 sync) — 차별화 핵심이므로 **첫 업데이트**. MVP에선 "공유 리스트 함께 보기"로 80% 대체.
- 음역대 측정 / 최고음 매칭 필터

### 7-4. 공유 모델 — 두 가지를 구분

| 모드 | 성격 | 동작 | 단계 |
|---|---|---|---|
| **플리 공유(fork)** | 비동기·솔로 | 링크/이미지로 공유 → 받은 사람이 "내 플리로 저장" = **독립 복제본**(내 라이브러리 귀속). 원본과 별개, 원작자 표기 | MVP |
| **그룹 세션(live)** | 실시간·공동 | 방코드/QR 입장 → 여러 명이 같은 셋리스트 함께 편집·순서 | v1.1 |

> **fork = "저장하면 내 것"**(스포티파이 플리 저장과 유사). **그룹 세션 = "같이 실시간 편집"**. 둘은 별개 기능이다.
> 익명 유저도 fork 가능(로컬 복제) → 가입 시 클라우드 동기화. 스샷 이미지는 클라이언트 렌더(서버비 0, 티켓 디자인 재활용). 곡 참조(song_id)+노트만 복제하므로 미디어/저작권 이슈 없음.

### 7-5. 발견(Discover) & 셀럽/큐레이션 플리

> **근거**: "아이돌 노래방 플리"는 트위터에서 이미 바이럴 (실제 사례: 팬 계정의 셀럽 '방구석 노래방 플리' = 2.4만 뷰·229 리트윗·368 좋아요). 그러나 지금은 **플레인 텍스트 스샷**이라 노래방 번호도, 저장도, 실사용도 안 됨. 우리 앱이 그 자리를 차지.

**구성 (기존 fork 인프라 위에)**
- **발견 탭**: 에디터·크리에이터·팬 큐레이션 + 인기/급상승 플리. 셀럽 플리 = verified/featured 공개 리스트.
- **저장 훅**: "○○의 노래방 플리" → 원탭 **내 플리로 저장(fork)** → 번호 포함이라 노래방에서 바로 사용.
- **뱃지**: 공식(제휴) / 에디터픽 / 팬제작 구분.

**단계**: MVP-light(에디터·팬 큐레이션, 비공식 라벨 시드) → v1.1+ 공식 아티스트/소속사 제휴(verified, §18 제휴 기둥과 연결).

**법적 주의(중요)**: 실존 인물명·사진 사용은 **퍼블리시티권·초상권** 이슈. 초기 안전 설계 = ① 셀럽 **사진 미사용**(텍스트·아바타만) ② **"팬 제작·비공식"** 명확 라벨 ③ 방송·인터뷰 등 **공개 발언 기반**(출처 표기). 상업적 전면 노출·수익화는 **공식 제휴 확보 후**. (§16)

## 8. 계산 로직 (Session Calculator)

### 8-1. 입력
- `songs[]` (담은 곡, N개), 곡별 `duration_min` (기본값 사용/조정 가능)
- `mode`: `coin_per_song` | `coin_per_time` | `room_per_time`
- `people` (인원, ≥1)
- 요금 파라미터 (사용자 입력 또는 프리셋)

### 8-2. 시간 추정
```
avg_min_per_song = 3.7 (기본, 조정 가능)   # 간주점프·취소 감안한 현실값
total_min = Σ song.duration_min  (없으면 avg 사용)
```
> 곡 길이는 iTunes `trackTimeMillis`로 힌트를 얻을 수 있으나, **노래방 버전 ≠ 음원 길이**(간주 포함/편집)라 기본 평균값이 더 안전. "정확 예측이 아니라 대략 감"임을 UI에 명시.

### 8-3. 모드별 금액 (정방향)
```
[코인·곡당]  total_cost = N × price_per_song            # 예: 500원/곡
             (묶음요금(X원에 Y곡) + 단품가 P 동시 존재 시 최저가 조합:
              total_cost = (floor(N/Y) × X) + min((N % Y) × P, X))
             # 남은 곡은 '단품 결제' vs '묶음 한 번 더' 중 저렴한 쪽 — 과다청구 방지
[코인·시간제] blocks = ceil(total_min / block_min)
             total_cost = blocks × block_price          # 예: 30분 5,000원
[룸·시간제]  hours = ceil(total_min / 60)
             total_cost = hours × hourly_rate           # 룸 단위, 인원 무관
             (서비스타임 옵션: +free_min)

per_person = total_cost / people
```

### 8-4. 역방향 (예산/시간 → 곡수) ★킬러
```
입력: budget(원) 또는 time_limit(분)
[곡당]  max_songs = floor(budget / price_per_song)
[시간]  max_songs = floor(time_limit / avg_min_per_song)
→ 리스트에서 우선순위 상위 max_songs곡 자동 선정, 나머지는 "다음에"
```
"1시간 잡았는데 26곡이면 다 못 부름 → 15곡만 추림"의 **의사결정을 대신 해줌**.

## 9. 데이터 전략 (곡 DB — 최대 운영 리스크)

### 9-1. 현황
- TJ·금영 **공식 공개 API 없음.** 기존 앱은 자체 크롤링+자체 DB(회색지대).
- 곡번호·곡명·가수는 개별로는 보호 안 되는 사실정보. 그러나 **DB 전량 복제**는 데이터베이스제작자권 침해 주장 영역(잡코리아 vs 사람인, 4.5억 배상).

### 9-2. 권장 전략 (리스크 '중→하')
```
① 시드 구축: 오픈소스/2차 소스로 초기 카탈로그
② 사용자 기여: "이 곡 없어요 → 번호 입력"으로 롱테일·누락 보정
③ 신곡 델타 수집: TJ/금영 신곡 목록만 주기 폴링 → DB에 없는 것만 추가
④ 통째 미러링 금지 + 앱 내 출처 고지 + 삭제요청 연락처
```

### 9-3. 신곡 갱신이 안 힘든 이유
- **음원 신곡 ≠ 노래방 신곡**: 번호 붙는 신곡은 훨씬 적고 배치(월별 정기)로 나옴. TJ/금영이 번호와 함께 직접 발표.
- 갱신 = 전체 재크롤링이 아니라 **델타만 append** (cron 하루 1회, 무료 티어).
- 곡에 번호가 없으면 노래방에서 부를 수도 없음 → **우리는 뒤처질 수가 없다.** 며칠 지연은 유저 체감 불가.
- **신곡만 경량 수집 = 법적 리스크도 낮춤** ("상당 부분 복제" 논점 약화).

## 10. 아키텍처 & 기술 스택 (로그인 없음 전제)

### 10-1. 2단계 전략
- **Phase A (검증, ~3주)**: **Next.js PWA + Supabase.** 로그인 없이 코어 루프(검색→담기→계산→티켓 링크 공유). URL이 곧 공유 수단, 카톡 바이럴 최강, 스토어 심사 없음.
- **Phase B (확장, +3~4주)**: 반응 있으면 **React Native(Expo) + Supabase** 네이티브(안드로이드 우선). 푸시, 온디바이스 곡 DB(오프라인), 실시간 룸.

### 10-2. 왜 Supabase
- Postgres `pg_trgm` + FTS + 초성 인덱스 컬럼으로 **검색을 DB단에서** 해결(Firebase 대비 결정적 이점).
- 실시간 룸: Supabase Realtime(무료: 동시 200접속·월 200만 메시지)로 초기 충분.
- **익명 인증(Anonymous Auth)** 지원 → 로그인 화면 없이 서버 동기화(§11).

### 10-3. 미디어 원칙
> **"재생/이미지는 남의 CDN, 우리는 URL만."** 앨범아트는 iTunes(애플 CDN) URL만 저장·전달. 오디오/미디어 자체 호스팅 절대 안 함 → 10만 MAU에서도 인프라비가 폭발하지 않음.

## 11. 신원 전략 — 익명 우선 + 지연 이메일 가입 (Progressive)

> 사용자 요구: 가입은 최대한 안 하고 싶다. **첫 진입은 무가입**, 계정이 꼭 필요한 기능(결제·다기기 동기화)에 도달했을 때만 가입(지연 가입 / lazy registration).

### 11-1. 3단계 신원 모델
| 단계 | 언제 | 방식 | 가능 기능 |
|---|---|---|---|
| **① 익명(기본)** | 첫 진입~ | 기기 로컬(IndexedDB) + 익명 세션 UUID(Supabase Anonymous). **가입 0** | 검색·담기·계산·티켓 생성·링크 공유(생성/열람)·방 입장·**기기 내 저장** |
| **② 이메일 가입(지연)** | 결제 / 다기기 동기화·백업 진입 시 | **Passwordless 이메일**(매직링크 or 6자리 OTP). **비밀번호 없음** | 구독 결제·엔타이틀먼트, 클라우드 동기화·백업, 다기기 |
| ③ 소셜(선택·후순위) | 편의 필요 시 | 카카오 등 | — 단 소셜 추가 시 **Apple 로그인 의무 발생** → 신중 |

익명 세션은 가입 시 그대로 **업그레이드**(로컬 데이터를 계정으로 이관) → 재입력 없이 자연스럽게 승격.

### 11-2. 가입 경계(Gate)
- **필요**: 구독 결제, 다기기 동기화/클라우드 백업, (선택) 공유물 소유권 주장.
- **불필요**: 검색·담기·계산·티켓 만들기·링크 공유 생성/열람·방 입장·**기기 내 저장**.
> ⚠️ "저장"은 익명으로도 **기기 안에는** 됨. 가입이 필요한 건 정확히는 **다기기 동기화·클라우드 백업**.

### 11-3. 왜 이메일-only(+Passwordless)인가
- **가입 체감 최소**: 이메일 입력 → 링크/코드. 비밀번호 없음 → 분실·재설정 CS 제거.
- **Apple 로그인 의무 회피**: Sign in with Apple(4.8)은 *소셜/서드파티* 로그인 사용 시 발동. **자체 이메일 인증만 쓰면 해당 없음** → 소셜 미도입 시 Apple 로그인 강제 없음.
- **결제·복구가 깔끔**: 결제·엔타이틀먼트가 이메일 계정에 귀속 → 기기 바꿔도 복구 쉬움(익명 결제의 복구 CS 문제 해소).

### 11-4. 트레이드오프 & 대응
| 트레이드오프 | 대응 |
|---|---|
| 익명 상태 기기 초기화 시 데이터 유실 | 가입 유도(동기화) + 익명 단계도 서버 백업 옵션 |
| 계정 생기면 계정 삭제 의무(Apple 5.1.1) | 앱 내 "계정·데이터 전체 삭제" 제공(간단) |
| 익명→가입 데이터 이관 | 익명 세션 UUID를 계정에 연결·병합 |
| 이메일 도달성 | 매직링크+OTP 병행, 재발송, 스팸함 안내 |
| 어뷰징/스팸 | 방/공유 링크 rate-limit·만료·신고, 라이선스 활성 기기 수 제한(예: 2대) |

- **국제화**: 한국어 우선. 숫자·통화 로케일.

- **제품 내 바이럴 루프**(최대 자산): 티켓 공유 링크 → 미설치자도 웹 열람 → "모여!".
- **ASO**: "노래방 번호/TJ/금영/애창곡" 키워드.
- **릴스/틱톡**: 노래방=챌린지 촬영 공간 트렌드 결합. "우리 90분 셋리스트" 티켓 캡처 공유 포맷.
- **커뮤니티**: 에브리타임, 디시 노래연습/노래방 갤러리, 케이팝 팬덤.
- **팬덤/셀럽 플리** ★: 아이돌·가수 노래방 플리를 번호 포함·저장 가능하게 → 트위터/케이팝 팬덤에서 이미 검증된 바이럴 포맷을 흡수. 최상위 유입 훅(§7-5).
- 유료 CPI(4천~1.5만원)는 광고 ARPU로 회수 불가 → **초기 배제**.

## 20. 개발 로드맵

| 단계 | 기간 | 산출물 |
|---|---|---|
| **주 0** | 1.5~2주 | 곡 DB 시드 + 신곡 델타 파이프라인, 출처/삭제요청 체계 |
| **Phase A (PWA MVP)** | 2~3주 | 검색+담기+계산기+티켓 공유 (로그인 없음). 지인 20~30명 베타 |
| **의사결정 게이트** | — | 공유 링크 클릭→방문 전환·주간 재방문 유의미하면 진행 |
| **Phase B (네이티브)** | 3~4주 | RN/Expo 안드로이드 + 앨범아트 + 히스토리 + 오프라인 DB |
| **v1.1** | +1.5~2주 | 실시간 그룹 큐(방코드·QR) + 플리 임포트 고도화 |
| **수익화** | 지표 후 | 라이트 구독 + 리워드 광고 |

> **지연 가입 효과**: 코어 루프(검색·담기·계산·티켓 공유)는 무가입이라 MVP에 인증 작업 없음. 소셜 로그인·Sign in with Apple은 이메일-only로 회피. 이메일 인증은 결제·동기화 단계(Phase B~수익화)에서만 붙이면 됨 → MVP가 빨라짐.

**MVP 정의(Phase A 종료 조건)**: 곡 검색 → 리스트 담기 → 세션 계산(코인 곡당/시간제 + 룸) → 티켓 링크 공유가 로그인 없이 웹에서 동작.
