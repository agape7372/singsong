# UNKNOWN_REGISTER.md — 제품 Unknown·가설 검증 장부

> **버전**: v1.2 · 2026-07-21
> **목적**: 구현 전에 숨은 가정과 미확정 결정을 드러내고, 각 항목을 증거 기반으로 닫는다.  
> **정본 경계**: 이 문서는 “아직 모르는 것과 검증 계획”의 장부다. 항목이 닫히면 최종 결정은 `FINAL_BLUEPRINT.md` 또는 `VISUAL_MOTION_DIRECTION.md`의 해당 정본에 반영하고, 이 문서에는 결정 링크와 증거만 남긴다.  
> **완료 원칙**: 코드가 동작한다는 사실만으로 제품 unknown은 닫히지 않는다. 통과 기준을 충족한 관찰·실측·계약·법적 근거가 있어야 한다.

---

## 0. 사용법

### 우선순위

- **P0**: 구현 방향·데이터 구조·공개 모델을 바꾸는 항목. 보수적 기본결정으로 local demo용 contract/test를 만들 수는 있지만, 명시된 검증시점 전까지 닫히지 않으면 해당 production 기능·외부 베타를 잠근다.
- **P1**: 공개 베타의 신뢰·성공 판정을 좌우하는 항목. 베타 시작 또는 베타 종료 게이트 전까지 닫는다.
- **P2**: 코어 가치 검증 후 판단할 확장 가설. P0/P1 결과 없이 선행 구현하지 않는다.

### 상태 표기

- **OPEN**: 검증 미시작
- **TESTING**: 검증 진행 중. 아래 체크박스는 아직 비워 둔다.
- **BLOCKED**: 외부 blocker 때문에 검증 중단. blocker와 다음 확인일을 상태 줄에 적는다.
- **CLOSED-PASS**: 통과 기준 충족. 상태 체크박스를 체크하고 증거·정본 반영 링크를 적는다.
- **CLOSED-FAIL**: 가설 기각 또는 기본결정 변경. 상태 체크박스를 체크하고 대체 결정·정본 반영 링크를 적는다.

### 갱신 규칙

1. 상태를 닫을 때 “느낌상 해결”을 금지한다. 검증 산출물 경로, 표본, 날짜, 담당자를 남긴다.
2. 통과 기준을 검증 도중 완화해야 한다면 먼저 변경 이유를 기록한다. 결과를 본 뒤 기준을 낮추지 않는다.
3. P0 기본결정이 바뀌면 PRODUCT_SPEC, UX_FLOWS, SCREENS, API_CONTRACT, ANALYTICS 등 해당 정본을 같은 변경에서 갱신한다.
4. 개인정보·공개범위·권리 문제는 실험군에 몰래 노출해 검증하지 않는다. 보수적 기본결정을 먼저 적용한 뒤 범위를 넓힌다.
5. 모든 수치 기준은 v1.0의 **사전 가설 게이트**다. 실제 베타 전에 제품 책임자가 표본과 함께 승인하거나 변경한다.

### U-P0-01 · Rank 1 — 곡 DB 권리·지속 가능한 데이터 소스

- **ID**: U-P0-01
- **질문**: TJ/KY 곡명·가수·번호를 어떤 출처와 권리 근거로 시드·갱신하며, 서비스가 커져도 같은 방식으로 운영할 수 있는가?
- **왜 숨겨졌나**: 개별 곡번호가 사실정보라는 점과 데이터베이스 전량 복제의 제작자권 위험이 섞여 있어 “회색지대지만 가능”으로 축약됐다. PRODUCT_SPEC §9의 시드 소스가 아직 구체적으로 결정되지 않았다.
- **영향**: **치명적**. 검색은 모든 핵심 여정의 입구다. 권리 근거가 없으면 출시 중단·데이터 삭제·재구축 비용이 발생하고, 데이터가 빈약하면 첫 사용 가치가 사라진다.
- **현재 기본결정**: 출처·라이선스·수집 범위가 문서로 확인되지 않은 데이터는 프로덕션 시드에 넣지 않는다. 통째 미러링은 금지하고, 허용 근거가 있는 초기 시드 + 사용자 제보 + 공식 신곡 델타만 검토한다.
- **검증 방법**:
  1. 후보 소스별 URL, 운영자, 라이선스/이용약관, 수집 방식, 필드, 갱신 주기를 표로 만든다.
  2. 각 소스가 전체 복제·상업적 이용·재배포를 허용하는지 법률 검토 메모를 남긴다.
  3. TJ/KY 삭제요청 접수·곡 단위 삭제·기존 공유 스냅샷 처리 절차를 리허설한다.
  4. 허용 근거가 모호한 후보는 독립 법률 자문 전까지 제외한다.
- **통과 기준**:
  - 프로덕션에 들어가는 모든 행이 출처·수집일·권리 근거로 추적 가능하다.
  - “허용 근거 미확인” 소스가 0개다.
  - 전체 DB 미러링에 의존하지 않고 초기 검색 골든셋을 충족할 수 있다.
  - 삭제요청 연락처·담당자·처리 절차·목표 처리시간이 확정되어 있다.
- **검증시점**: 데이터 마이그레이션·시드 스크립트 구현 전. 닫히지 않으면 프로덕션 데이터 파이프라인 구현을 잠근다.
- **외부 blocker**: 데이터 제공자의 약관/라이선스 명확성, 필요 시 국내 데이터베이스권 법률 자문, 실제 공식 신곡 목록 접근성.

### U-P0-02 · Rank 2 — 공유 공개성·색인·payload 최소화

- **ID**: U-P0-02
- **질문**: 공유 티켓은 “링크를 가진 사람만 보는 unlisted 문서”인가, 검색엔진과 API에서 발견 가능한 공개 게시물인가? 정확히 어떤 필드가 공개되는가?
- **왜 숨겨졌나**: UX 카피는 “링크를 아는 사람 누구나”라고 설명하지만 SCREENS는 공유 페이지 색인을 허용한다. API_CONTRACT의 public SELECT 정책은 exact slug 조회와 전 행 열람을 구분하지 않으며, payload에는 myKey·memo·tags가 포함된다.
- **영향**: **치명적**. 사용자가 비공개라고 생각한 메모·키·리스트가 검색·대량수집될 수 있고, 공개범위 오고지는 신뢰·개인정보·운영 리스크로 직결된다.
- **현재 기본결정**: MVP 공유는 **unlisted + noindex**다. 브라우저는 Supabase를 직접 호출하지 않고 Next BFF가 exact slug를 처리한다. 기본 payload는 곡 제목·가수·TJ/KY 번호·순서·versioned 계산 입력과 범위 요약만 포함한다. myKey·memo·tags·device UUID는 P0 schema 자체에 없다.
- **검증 방법**:
  1. PUBLIC/anon/authenticated로 catalog/shared/rate table·view·sequence·function의 직접 조회/쓰기/실행과 pagination/order/select=*를 시도한다.
  2. Next BFF의 정확한 slug 조회만 성공하고 목록·범위 조회 route가 없으며 browser bundle에 Supabase credential이 없음을 검사한다.
  3. 생성된 공유 행과 네트워크 payload를 확인해 비공개 필드가 없는지 검사한다.
  4. robots meta·robots.txt·sitemap·OG crawler 경로를 점검한다.
  5. 5명에게 공개범위 카피를 보여주고 누가 볼 수 있다고 이해했는지 되묻는다.
- **통과 기준**:
  - anon 클라이언트의 전 행/범위 조회가 거부된다.
  - exact slug 조회만 성공한다.
  - 공유 payload에 myKey·memo·tags·device UUID가 0건이다.
  - 공유 페이지는 noindex이며 sitemap에 들어가지 않는다.
  - 5명 중 5명이 “링크가 전달된 사람은 볼 수 있지만 검색 결과에는 나오지 않는다”고 정확히 설명한다.
- **검증시점**: shared_plans DDL/ACL과 BFF 공유 스키마 구현 전. 공개 테스트 데이터라도 이 결정 전에는 외부 배포하지 않는다.
- **외부 blocker**: 없음. 필요 시 개인정보·공개고지 문구 법률 검토.

### U-P0-03 · Rank 3 — 핵심 사용자 occasion·JTBD

- **ID**: U-P0-03
- **질문**: 가장 먼저 만족시킬 사용자는 혼코노 기록자, 2~4인 소그룹 주최자, 룸노래방 모임 중 누구이며, 앱을 실제로 여는 순간은 언제인가?
- **왜 숨겨졌나**: PRODUCT_SPEC은 2030 혼코노와 소그룹, 3040 룸 사용자를 한 타깃으로 묶었다. 그러나 키·메모·히스토리의 JTBD와 계산·1인당·공유의 JTBD는 다르다.
- **영향**: **치명적**. 타깃이 정해지지 않으면 홈 위계·기본 계산 모드·티켓 정보·MVP 범위·성공지표가 모두 흔들린다.
- **현재 기본결정**: 초기 사용자는 **향후 7일 안에 2~4인 노래방 약속을 제안하거나 준비하는 한 명의 주최자**다. 혼코노 기록과 3040 룸은 후순위다.
- **검증 방법**:
  1. 최근 30일 안에 노래방에 간 12명 이상을 인터뷰한다.
  2. “그럴 것 같다”가 아니라 실제 단톡·메모·검색 기록과 마지막 방문 준비 과정을 회고한다.

  4. 종이/클릭 프로토타입으로 검색→담기→계산→공유 과업을 관찰한다.
- **통과 기준**:
  - 한 개의 주 사용자·한 개의 시작 occasion·한 개의 핵심 결과를 한 문장으로 확정한다.
  - 소그룹 주최자 6명 중 4명 이상이 최근 실제 상황에서 시간·비용·곡 조율 중 최소 2개를 경험했다.
  - 5명 이상이 도움 없이 2분 안에 프로토타입의 첫 계산 결과에 도달한다.
  - 제외할 사용자와 제외 이유가 PRODUCT_SPEC에 명시된다.
- **검증시점**: 홈 IA·기능 범위·마이크로카피 잠금 전.
- **외부 blocker**: 최근 실제 이용자를 모집할 채널, 인터뷰 동의, 실제 대화 기록을 보여줄 수 있는 참여자.

### U-P0-04 · Rank 4 — iOS·Kakao·PWA 저장소 섬과 import handoff

- **ID**: U-P0-04
- **질문**: 카카오 인앱 브라우저에서 “내 플리로 저장”한 데이터가 Safari/Chrome/PWA에서도 보이는가? 보이지 않는다면 사용자는 어디에 저장됐다고 이해하는가?
- **왜 숨겨졌나**: PLATFORM_NOTES는 카카오의 Web Share·다운로드 제약을 다루지만, WebView별 IndexedDB 격리와 가져오기 후 외부 브라우저/PWA로 이동하는 데이터 handoff는 다루지 않는다.
- **영향**: **치명적**. 핵심 바이럴 유입의 대부분이 카카오라면 fork가 일시적 WebView 저장소에 갇혀 다음 방문에 사라질 수 있다. “저장했어”라는 성공 카피가 거짓이 된다.
- **현재 기본결정**: 저장소 공유가 실측으로 증명되기 전에는 카카오 WebView 안의 `공유 스냅샷 가져오기`(레거시 명칭 `fork`)만으로 완료 처리하지 않는다. 외부 브라우저에서 전체 공유 URL을 다시 열거나 설치 PWA의 `/import`에 URL/slug를 붙여넣어 가져오기를 끝낸 뒤 완료로 친다. 별도 one-time token은 이 단순 경로가 실패할 때만 검토한다.
- **검증 방법**:
  1. iOS 카카오→Safari import, iOS 카카오→설치 PWA import, Android 카카오→Chrome import, Android 카카오→설치 PWA import를 실기기로 검증한다.
  2. 카카오 앱 종료·재실행, OS 앱 종료, 다음날 재접근까지 확인한다.
  3. 격리가 재현되면 링크 복사→외부 브라우저 또는 설치 PWA `/import` 붙여넣기 프로토타입을 구현해 같은 exact snapshot을 가져오는지 검증한다.
  4. handoff 실패·만료·중복 탭·뒤로가기·앱 미설치 상태를 시험한다.
- **통과 기준**:
  - 4개 경로 모두에서 저장한 리스트가 사용자가 다음에 실제 사용할 브라우저/PWA에 남는다.
  - 동일 slug의 중복 import가 발생하지 않는다.
  - handoff 실패 시 저장 성공 토스트가 나오지 않고 복구 행동이 보인다.
  - 5명 중 5명이 “어디에 저장됐고 다음에 어디서 여는지” 설명할 수 있다.
- **검증시점**: import 데이터 모델·성공 카피 구현 전 기술 스파이크. 늦어도 공개 베타 전.
- **외부 blocker**: 최신 iOS/Android 실기기, 카카오톡 계정·최신 앱, HTTPS 배포 도메인, PWA 설치 가능 빌드.

### U-P0-05 · Rank 5 — 공동선곡 JTBD vs 읽기전용 공유

- **ID**: U-P0-05
- **질문**: 주최자가 만든 읽기전용 티켓과 독립 가져오기만으로 친구들의 “무엇을 부를지 합치기” 문제를 충분히 해결하는가?
- **왜 숨겨졌나**: 문제 정의는 중구난방인 곡을 합쳐 순서·시간을 정하는 일을 말하지만, MVP는 원본과 독립된 가져오기이고 실시간 그룹 편집은 P1로 미뤄졌다.
- **영향**: **높음**. 핵심 통증과 실제 제공 기능이 다르면 공유는 예쁜 결과물에 그치고 그룹 계획 도구로 쓰이지 않는다.
- **현재 기본결정**: MVP의 약속을 “한 주최자의 계획을 명확히 전달”로 제한한다. 공동 합의가 필수로 관측되면 실시간 룸보다 작은 **비동기 곡 제안/투표**를 먼저 검토한다.
- **검증 방법**:
  1. 실제 2~4인 단톡방 5개에 주최자 티켓을 공유한다.
  2. 수신자가 읽기만 하는지, 곡 추가·삭제·투표·담당자 지정·중복곡 요청을 하는지 관찰한다.
  3. 읽기전용안과 “곡 하나 제안하기” 프로토타입을 비교한다.
- **통과 기준**:
  - 5개 그룹 중 4개 이상이 읽기전용 티켓만으로 약속 결정 또는 현장 준비를 완료한다.

### U-P0-08 · Rank 8 — 공유 철회·만료·takedown

- **ID**: U-P0-08
- **질문**: 계정 없는 공유자가 실수로 공개한 티켓을 어떻게 철회하며, 운영자는 신고를 얼마나 빨리 처리하고 캐시된 OG까지 어떻게 무효화하는가?
- **왜 숨겨졌나**: 불변 스냅샷 단순화를 위해 anon UPDATE/DELETE를 모두 막고 운영자 수동 takedown만 남겼다. 그러나 사용자 본인 철회, 만료, OG 캐시, 기기 변경은 정의되지 않았다.
- **영향**: **높음**. 잘못 공유한 내용이 장기간 남고, 신고 대응이 늦으면 신뢰·법무 문제가 생긴다.
- **현재 기본결정**: 공유 생성 시 브라우저에만 저장되는 revocation secret을 만들고 서버에는 hash만 저장한다. 사용자는 같은 기기에서 철회할 수 있다. P0 기본 만료는 30일이며, 베타 증거로만 변경한다. 만료 없는 영구 공개는 채택하지 않는다.
- **검증 방법**:

  3. 신고 이메일에 slug가 없거나 잘못된 경우의 운영 절차를 작성한다.

  - 철회 후 origin 일반 조회와 origin OG endpoint는 즉시 콘텐츠를 반환하지 않는다.
  - 제3자 preview cache의 실제 잔존 시간과 purge 가능 여부를 기록하고, 통제할 수 없으면 사용자 고지·운영 runbook과 `BLOCKED + EXTERNAL_AUTHORITY` 증거로 남긴다.
  - secret 원문은 서버·로그·analytics에 저장되지 않는다.
  - 운영자 신고 접수 주소·담당자·목표 SLA가 확정된다.
- **검증시점**: shared_plans 스키마 확정 전. 늦어도 외부 공유 베타 전.

### U-IMP-01 — DB commit 뒤 응답 유실·Turnstile single-use·slug 복구

- **질문**: share row가 commit된 뒤 HTTP 응답만 사라지거나 Siteverify와 DB 사이가 끊겨도 같은 요청을 중복 row·quota 소비·다른 URL 없이 복구할 수 있는가?
- **왜 숨겨졌나**: hash-only random slug는 역복구가 불가능하고 Turnstile token은 5분·single-use다. happy-path POST 한 번으로는 모순이 보이지 않는다.
- **현재 기본결정**: versioned HMAC으로 idempotency key에서 slug를 결정적으로 파생한다. committed row 조회를 Siteverify/quota보다 먼저 한다. Siteverify UUID는 서버가 domain-separated SHA-256으로 `share idempotency hash + token hash`에 안정적으로 bind하므로 restart/key rotation 없이 같은 retry UUID를 유지하고 cross-share cached success 재사용을 막는다. slug key secret은 마지막 해당-version row 뒤 최소 45일 보존한다. revoke/expiry 관측 transaction은 reservation을 즉시 terminalize하며 payload 없는 hash reservation을 남겨 cleanup 전후 오래된 POST가 폐기 URL을 부활시키지 못하게 한다.
- **필수 증거**: 검증 전, Siteverify 응답 유실/성공 뒤 DB 전, DB commit 뒤 HTTP 응답 전 fault injection; key rotation·secret 누락 startup gate; 동일 raw slug·row 1개·quota 1회; active row cleanup 뒤 같은 POST replay가 generic 409이고 exact URL은 계속 unavailable임을 확인.
- **상태**: [ ] CONTRACT_LOCKED / IMPLEMENTATION_UNVERIFIED

### U-IMP-02 — 동일 revision 티켓의 reload·다중 탭 정체성

- **질문**: 같은 `{planId,revision}`을 두 탭에서 동시에 열거나 reload·재공유해도 artworkSeed·payload·fingerprint·PNG/OG가 동일한가?
