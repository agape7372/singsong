## 1. 우선순위 등급 정의

코덱스프롬프트.txt §5.16을 그대로 채택한다(값 복사가 아니라 동일 정의 그대로 사용 — 이 체계의 정본은 코덱스프롬프트.txt이며 이 문서는 재수록만 한다).

| 등급 | 정의 |
|---|---|
| **Critical UX** | 사용자가 핵심 작업(담기→계산→티켓→공유)을 완료할 수 없거나, 데이터·개인정보와 관련된 심각한 오작동 |
| **High** | 다수 사용자가 핵심 흐름에서 이탈하거나 중대한 실수(예: 위조 payload 실행, 잘못된 금액 표시)를 할 가능성이 높은 문제 |
| **Medium** | 사용자가 반복적으로 혼란·불편을 경험할 가능성이 높은 문제 |
| **Low** | 일관성·가독성·완성도를 개선하는 문제 |
| **Observation** | 실제 사용자 테스트 또는 기획 확인이 필요한 사항(`기획 확인 필요` 표기와 연동) |

싱송 맥락 보정: "핵심 작업"은 BUILD_PLAN §1 MVP 포함 범위(검색→담기→계산→티켓→공유→fork)를 기준으로 한다. 로그인·결제·발견 탭은 범위 밖이므로 그 영역의 문제는 Critical이 될 수 없다(있다면 "범위 밖" 주석과 함께 Observation으로 강등).

## 4. 자가감사 채점표 양식

`ONESHOT_MASTER.md` Phase 8.5 필수, 코덱스 vs 오푸스 비교 채점에 사용.

### 4-1. 배점 (합 100)

| 카테고리 | 배점 |
|---|---|
| C1 사용자 여정 | 10 |
| C2 IA·내비게이션 | 6 |
| C3 시각 위계 | 6 |
| C4 디자인 일관성 | 6 |
| C5 반응형·모바일 | 8 |
| C6 접근성 | 10 |
| C7 폼·입력 | 6 |
| C8 상태 표현 | 10 |
| C9 인터랙션 피드백 | 4 |
| C10 문구·현지화 | 6 |
| C11 신뢰·안전 | 8 |
| C12 체감 성능 | 4 |
| C13 FE-BE 계약 일치 | 8 |
| C14 보안 | 8 |
| **합** | **100** |

가중치 근거: MVP 핵심 여정(C1)·상태 표현(C8)·접근성(C6)에 최고 배점. 서버 계약(C13)·보안(C14)은 위조 payload·RLS 우회가 곧 Critical로 이어지므로 8점. 체감 성능(C12)은 실측 불가 항목이 많아 낮게(4점).

### 4-2. 감점 규칙

- 해당 카테고리에 **Critical UX 1건 이상** → 그 카테고리 점수 **0점** (다른 결함 무관).
- **High 1건** → 카테고리 배점의 **50% 차감**(반올림). High 2건째부터는 카테고리 0점.
- **Medium 1건** → 카테고리 배점의 **15% 차감**. 최대 3건까지 누적, 이후는 0점 처리.
- **Low** → 건당 5% 차감, 카테고리 배점의 최대 20%까지만.
- **Observation** → 감점 없음. 별도 "확인 필요" 목록에 기록.
- 해당 화면이 "—"(해당없음)인 셀은 채점에서 제외하고 나머지 셀 평균으로 카테고리 점수 산정.

### 4-3. 기록 형식

한 줄 = 한 문제. 표 형식으로 append:

```
| 문제 제목 | 등급 | 화면 | 파일 경로 근거 | 수정 여부 |
|---|---|---|---|---|
| 예: 검색 IME 조합 중 Enter로 빈 검색 실행 | High | /search | app/search/page.tsx:42 | 수정함 |
```

- "파일 경로 근거"는 반드시 실제 코드 파일:줄번호. 근거 없는 항목은 Observation으로 강등.
- "수정 여부": `수정함` / `수정 보류(사유)` / `해당없음(설계상 의도)`.
- Critical·High는 **즉시 수정 후 재채점**해야 Phase 8.5 완료조건 충족(`ONESHOT_MASTER.md` Phase 8.5).
