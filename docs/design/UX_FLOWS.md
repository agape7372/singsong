## 1. 핵심 여정 5개

### 여정 ① 첫 방문 → 첫 담기 (온보딩 없이 빈 상태가 온보딩)

- **사용자 목표**: 뭔지 잘 모르는 채로 들어왔지만, 곧바로 "아 이거 노래방 계획 짜는 거구나"를 이해하고 첫 곡을 담는다.
- **진입점**: 공유 링크 없이 앱 URL/PWA 아이콘으로 직접 옴(오가닉·ASO·검색), 또는 §여정⑤에서 fork까지 마친 뒤 자연 유입.

**단계 다이어그램**
```
앱 진입 → 홈(빈 상태=온보딩) → "곡 담으러 가기" 탭 → /search
  → 검색 → 첫 곡 담기 → 자동으로 리스트 생성(오늘의 플리) → 홈 복귀(수동)
  → 계산 카드 처음 등장(기본 프리셋 값으로 즉시 결과 표시)
```

- **단계별 마찰 지점과 설계 대응**
  - 마찰: "이 앱이 뭘 해주는 건지" 첫 화면에서 즉시 안 와닿을 수 있음 → 대응: 홈 빈 상태 카피가 가치 제안을 한 문장으로 압축(〔문구: MICROCOPY §홈-첫방문〕), 가입 절차·튜토리얼 슬라이드 자체가 없어 마찰 지점 수를 원천적으로 줄임(PRODUCT_SPEC §7-3 명시 제외 항목).
  - 마찰: 검색 후 담기만 하고 홈으로 안 돌아오면 계산 결과를 못 봄(가치 체감 실패) → 대응: 담기 직후 Toast에 "N곡 담김 · 홈에서 계산 보기" 계열 넛지(〔문구: MICROCOPY §담기토스트〕), TabBar가 항상 보여 홈 복귀 마찰이 낮음.
  - 마찰: 첫 진입에 계산 파라미터(요금)를 뭘 넣어야 할지 모름 → 대응: 코인노래방 흔한 기본값 프리셋이 이미 선택된 채로 시작(BUILD_PLAN §11-3 프리셋), 사용자가 아무것도 안 눌러도 그럴듯한 숫자가 뜬다.
- **중간 이탈·복구**
  - 오프라인 전환: 검색만 서버 필요(SCREENS §4-4). 오프라인 상태로 진입해도 홈은 "리스트 없음" 빈 상태로 정상 렌더, 검색 진입 시에만 오프라인 안내가 뜬다.
  - 뒤로가기: 검색 화면에서 담기 없이 뒤로가면 그냥 빈 홈으로 복귀 — 손해 없음, 재유입 허들 낮음.
  - 새로고침: Dexie가 있으므로 담은 것까지는 유지, 아직 안 담았으면 처음부터.
- **성공 지표(계측 이벤트)**: `app_open` → `song_search` → `song_added` 로 이어지는 전환율. `calc_changed`가 최초 발생하면(자동 프리셋도 최초 1회 계산은 이벤트로 침) 가치 체감 도달로 간주.
- **엣지케이스**
  1. 검색은 했지만 원하는 곡이 DB에 없음(SCREENS §4-4 빈 결과) → 이 여정이 여기서 끊길 수 있음. 대응은 여정②로 이관(사용자 기여 경로는 MVP 범위 밖, PRODUCT_SPEC §9-2 참조 포인터만).
  2. 앱을 껐다 다음날 다시 켜서 이어서 담음 → `return_visit` 계측, 리스트는 그대로 있어 마찰 없음.
  3. 사파리 프라이빗 모드 등 IndexedDB 접근 실패 → SCREENS §3-4 에러 상태로 즉시 하강, 첫인상이 나빠지는 리스크(§미결).

### 여정 ② 검색 → 담기 반복

- **사용자 목표**: 노래방에서 부를 곡을 생각나는 대로 계속 검색해서 쌓는다(한 세션에 여러 번 반복).
- **진입점**: 홈에서 "곡 담으러 가기", 또는 TabBar '검색' 직접 탭(이미 곡이 있는 상태에서 추가).

**단계 다이어그램**
```
/search 진입 → 검색어 입력(디바운스) → 결과 목록
  → 담기(＋) 반복 N회 → (선택) 다시 검색어 지우고 새 검색 → 반복
  → 만족하면 뒤로가기 or TabBar '홈'
```

- **단계별 마찰 지점과 설계 대응**
  - 마찰: 매번 담을 때마다 화면이 리스트로 이동하면 연속 검색이 끊김 → 대응: 담기는 **화면 전환 없이** SongRow 내 상태만 바뀜(SCREENS §4-3), 검색은 `/search`에 머문 채 여러 곡 연속 담기가 가능.
  - 마찰: 한글 IME 조합 중 Enter로 검색이 잘못 트리거되거나 결과가 깨짐 → 대응: `isComposing` 상태를 존중하는 입력 처리(엔지니어링 몫, PLATFORM_NOTES 정본 — 이 문서는 여정 관점 포인터만).
  - 마찰: 이미 담은 곡을 또 검색해서 중복으로 담을까 걱정 → 대응: 이미 담긴 곡은 SongRow가 "담김" 상태로 시각적 구분(SCREENS §4-3), 재탭 시 제거 토글이라 중복 담기 자체가 불가능한 구조.
  - 마찰: 검색 결과가 없으면 그 자리에서 막힘 → 대응: 빈 결과 카피가 다른 표기(초성/번호/원어)로 재시도를 유도(〔문구: MICROCOPY §검색-결과없음〕).
- **중간 이탈·복구**
  - 오프라인 전환: 검색 도중 네트워크가 끊기면 SCREENS §4-4 오프라인 안내로 전환, 그때까지 담은 곡은 이미 로컬에 반영돼 있어 유실 없음.
  - 뒤로가기: 언제든 안전 — 담은 것은 이미 커밋된 로컬 상태라 "취소" 개념이 없음(장바구니처럼 미확정 상태를 갖지 않는다).
  - 새로고침: 검색어·스크롤은 초기화되지만 담긴 곡은 유지.
- **성공 지표**: `song_search` 대비 `song_added` 비율(검색이 실제 담기로 이어지는가), 한 세션 내 `song_added` 반복 횟수.
- **엣지케이스**
  1. 번호로 검색했는데 형식은 맞지만(예: 5자리 숫자) 존재하지 않는 곡 → 결과 없음과 동일 처리(SCREENS §4-4).
  2. 같은 곡을 서로 다른 리스트에 담고 싶음 → 리스트 전환은 홈에서만 가능(SCREENS §3-3 Chip), `/search`는 항상 "현재 리스트"에 담는다 — 다른 리스트에 담으려면 홈에서 리스트를 먼저 전환 후 재검색해야 함(마찰이지만 MVP는 단일 활성 리스트 개념으로 단순화, §미결).
  3. 결과가 50건 상한(API_CONTRACT §2-1)에 걸릴 만큼 흔한 검색어(가수명만 입력 등) → 상한 안에서 정렬 계약(번호 정확일치>전방일치>유사도)이 우선순위를 대신 정해주므로 원하는 곡이 보통 상위에 옴.

### 여정 ③ 계산 · 역계산 (모드 전환 포함)

- **사용자 목표**: "이 곡들 다 부르면 얼마 나와?"에서 "1시간/3만원이면 몇 곡 부를 수 있어?"까지 왔다갔다 확인한다.
- **진입점**: 홈의 세션 계산 카드(곡을 1곡이라도 담은 뒤부터 노출).

**단계 다이어그램**
```
홈(계산 카드 노출, 기본 프리셋) → 모드 선택(코인·곡당/코인·시간제/룸·시간제)
  → 인원 조정 → 요금 프리셋 선택 or 직접입력 → 결과(총시간·총금액·1인당) 실시간 갱신
  → (선택) 역계산 펼치기 → 예산/시간 슬라이더 조작 → "가능 곡수" 확인
  → 티켓 만들기로 진행 or 리스트로 돌아가 곡 조정
```

- **단계별 마찰 지점과 설계 대응**
  - 마찰: 3가지 모드·요금 파라미터가 처음엔 뭔지 헷갈릴 수 있음(코인 곡당 vs 시간제 vs 룸) → 대응: SegmentedControl 라벨을 노래방에서 실제 쓰는 표현 그대로(친구톤, MICROCOPY 정본), 기본 선택값은 가장 흔한 모드(코인·곡당)로 프리셋.
  - 마찰: 값을 바꿀 때마다 로딩처럼 느껴지면 "계산기가 무겁다"는 인상 → 대응: 계산은 순수 클라 함수라 지연 없이 즉시 갱신, 숫자 변화는 카운트업 모션으로 "실시간으로 반응한다"는 느낌을 강화(BUILD_PLAN §14).
  - 마찰: 역계산이 정방향 계산과 뭐가 다른지 개념 이해 필요 → 대응: 기본은 접혀 있고(§SCREENS 3-2) 필요한 사람만 펼치는 disclosure 패턴, 펼친 순간 카피가 "예산/시간 넣으면 몇 곡 되는지 알려줄게" 식으로 즉시 설명(§PRODUCT_SPEC 8-4 "킬러" 기능이므로 발견은 쉽게, 강제는 안 함).
  - 마찰: N=0(곡 없음)일 때 계산 카드가 에러처럼 보일까봐 → 대응: SCREENS §3-4에 명시한 대로 카드 진입 자체를 곡 1개 이상부터 허용, 0곡 상태는 계산 카드가 아예 안 뜬다(에러 회피를 UI 노출 자체를 막는 방식으로 해결).
- **중간 이탈·복구**
  - 오프라인 전환: 전부 로컬 계산이라 **영향 없음**(§4 오프라인 여정 참조).
  - 뒤로가기: 계산 카드는 홈에 상주하므로 "뒤로가기로 계산이 날아간다"는 개념 자체가 없음.
  - 새로고침: **유지된다(확정)** — 모드·인원·요금의 마지막 유효값은 Dexie `settings`(`lastMode`·`lastPeople`·`pricePreset`)에 자동 영속되어 다음 방문의 초기값이 된다(ARCHITECTURE §2-1). 역계산 슬라이더 값만 휘발(탐색용 입력이라 저장 안 함).
- **성공 지표**: `calc_changed` 발생 횟수(모드·인원·요금·역계산 조작 총합), 그중 역계산 슬라이더 조작 비율(킬러 기능 실사용률).
- **엣지케이스**
  1. 인원을 곡 수보다 훨씬 크게 설정(예: 30명) → `perPerson`이 매우 작아짐, 에러 아님(API_CONTRACT §4 `people` 상한 30과 정합).
  2. 묶음요금(X원에 Y곡) 입력 중 Y=0 등 비정상 값 → 입력 단계에서 최소값 제약으로 차단(계산 로직 자체의 방어는 BUILD_PLAN §6 포인터).
  3. 역계산 결과 곡수가 현재 담은 곡보다 많음(다 불러도 시간/예산이 남음) → "다 불러도 남아" 계열 카피로 양수 프레이밍(〔문구: MICROCOPY §역계산-여유〕).

### 여정 ④ 티켓 생성 → 공유(이미지/링크/카톡 3경로)

- **사용자 목표**: 계획한 세션을 "짜잔" 하고 보여줄 수 있는 형태로 만들어서 친구들에게 보낸다.
- **진입점**: 홈의 "티켓 만들기" CTA(곡 1곡 이상, 계산 결과가 이미 나와 있는 상태).

**단계 다이어그램**
```
홈 → 티켓 만들기 탭 → /ticket/[id] (TicketCard 발권 연출)
  → 3경로 중 선택:
    (a) 이미지 저장 → PNG 다운로드 → 끝(인스타 스토리 등 수동 공유)
    (b) 링크 공유 → 스냅샷 서버 저장 → slug 발급 → Web Share/클립보드 → 끝
    (c) 카톡 공유 → (b)와 동일 스냅샷 경로 재사용 → Kakao 공유 시트 → 끝
```

- **단계별 마찰 지점과 설계 대응**
  - 마찰: "공유하면 이게 공개되는 건가?" 불안 → 대응: 링크/카톡 경로를 누르기 **전에** 공개범위 고지가 이미 눈에 보이는 위치에 있어야 함(신뢰 UX §2 상세). 이 여정 문서에서는 고지 타이밍만 규정하고 문구는 MICROCOPY 몫.
  - 마찰: 이미지 저장이 "진짜 저장됐는지" 확신이 안 섬(다운로드 UX는 브라우저마다 다름) → 대응: 저장 완료 Toast로 명시적 확인(SCREENS §6-5).
  - 마찰: 링크 공유가 네트워크 문제로 실패했는데 사용자는 "카톡에 보냈다고 착각"할 위험 → 대응: 실패 시 Toast로 명확히 실패를 알리고, 성공 전까지 공유 시트를 열지 않는다(성공 확인 후에만 다음 단계 진행 — 순서 보장).
  - 마찰: 3경로 중 뭘 눌러야 할지 고민(선택 마비) → 대응: 버튼 순서·위계로 가장 캐주얼한 경로(카톡)를 시각적으로 우선하되(타깃이 카톡 바이럴을 최우선으로 두는 PRODUCT_SPEC §1), 나머지 두 경로도 동등하게 접근 가능하게 배치(강제 없음).
- **중간 이탈·복구**
  - 오프라인 전환: 이미지 저장은 유지, 링크/카톡 공유는 SCREENS §6-4대로 비활성화 + 안내. 온라인 복귀 시 자동 재활성.
  - 뒤로가기: 공유 진행 중 뒤로가도 이미 시작된 서버 요청은 계속 진행되도록 하되, 결과 알림을 놓치지 않게 하는 처리가 §미결(전역 Toast 큐 필요).
  - 새로고침: 아직 slug가 안 만들어진 상태에서 새로고침하면 그 시도는 사라지고 처음부터(부작용 없음 — 서버에 중간 상태가 안 남는 구조, API_CONTRACT §1-3 insert는 원자적).
- **성공 지표**: `ticket_created` → `share_created` 전환율(경로 무관 합산, 경로별 세부는 ANALYTICS 속성 스키마 몫).
- **엣지케이스**
  1. 곡이 100개를 넘는 대형 리스트로 링크 공유 시도 → API_CONTRACT CHECK(§1-3) 위반을 클라가 선검증으로 막고 "곡이 너무 많아" 안내(SCREENS §6-4).
  2. 카톡 앱이 없는 환경(데스크톱 브라우저 등)에서 카톡 공유 탭 → Kakao SDK가 웹 공유 대체 동작을 제공하는지는 실측 필요(§미결).
  3. 같은 티켓을 여러 번 다른 사람에게 나눠 공유(단톡방 A, 단톡방 B) → **동일 slug 재사용(확정)** — 티켓 화면을 벗어나지 않는 한 링크/카톡 재공유는 같은 스냅샷·같은 URL이다(SCREENS §6-4 확정 정책). 두 단톡방의 열람이 같은 slug의 `share_viewed`로 집계된다.

### 여정 ⑤ 링크 수신 → 열람 → fork → 재공유 (바이럴 루프)

- **사용자 목표**(받는 사람 시점): 친구가 보낸 티켓을 보고 "오 이거 좋다"에서 "나도 이걸로 가야지"까지, 설치 없이 바로 확인하고 저장한다.
- **진입점**: 카톡/링크로 받은 `/s/[slug]` URL. **이 여정 대부분의 참여자는 앱을 처음 접하는 미설치자**다(딥링크가 무설치 첫 진입점, PRODUCT_SPEC §1 최대 성장 엔진).

**단계 다이어그램**
```
카톡 인앱 브라우저(주로) or 일반 브라우저로 /s/[slug] 진입
  → SSR 렌더 완료(로그인 요구 없음) → 티켓+곡목록 열람
  → 공개범위 고지 확인 → "내 플리로 저장(fork)" 탭
  → 로컬 Dexie에 복제 생성(forkedFrom 기록) → 홈으로 이동 → (이제 이 사람이 여정①의 "곡 있는 상태"로 시작)
  → 이후 이 사람이 자기 계산을 돌리고 → 여정④로 다시 진입 → 재공유
```

- **단계별 마찰 지점과 설계 대응**
  - 마찰: 카톡 인앱 브라우저는 기능 제약이 있을 수 있음(iOS 특이사항, PLATFORM_NOTES 정본 포인터) → 대응은 엔지니어링 몫이나, 여정 관점에서는 **fork 버튼 하나로 끝나는 최소 동작**만 요구하도록 이 화면을 설계해 인앱 브라우저 제약의 영향 범위를 최소화(SCREENS §7이 읽기전용+버튼 하나 구조인 이유).
  - 마찰: "이거 저장하면 뭐가 되는 거지?"(fork 개념이 낯설 수 있음) → 대응: 버튼 라벨 자체가 결과를 설명("내 플리로 저장" — 저장하면 내 것이 된다는 스포티파이식 익숙한 멘탈모델, PRODUCT_SPEC §7-4).
  - 마찰: 미설치자가 fork 이후 "그래서 이제 뭘 하지?" 방향을 잃음 → 대응: fork 성공 후 **자동으로 홈 이동**(SCREENS §7-6), 홈은 이미 곡이 채워진 상태로 시작하므로 여정①의 "빈 상태 온보딩"을 건너뛰고 바로 여정③(계산)으로 자연 진입 — 빈 상태를 못 보고 곧장 가치를 체감.
  - 마찰: 이 플리가 진짜 믿을 만한가 → 대응: 티켓 카드가 곡목록·계산 결과를 그대로 보여주는 것 자체가 신뢰 근거(내용이 전부 공개됨). 익명 구조라 작성자 신원 표기는 없다(§3-2 확정) — "보낸 친구가 곧 신뢰의 출처"인 카톡 전달 맥락에 의존.
- **중간 이탈·복구**
  - 오프라인 전환: 이 화면은 SSR 필수라 오프라인이면 애초에 못 들어옴(SCREENS §7-4) — 여정이 시작조차 못 하는 게 유일한 실패 모드, 복구는 온라인 대기뿐.
  - 뒤로가기: fork 전 뒤로가기는 앱 밖(카톡)으로 나가는 정상 이탈 — 손해 없음(아직 아무것도 안 바뀜). fork 후 뒤로가기는 SCREENS §7-6에서 홈 이동으로 대체되어 이 화면에 다시 안 옴.
  - 새로고침: fork 전 새로고침은 같은 slug로 다시 SSR 렌더될 뿐 무해. fork 후에는 이미 홈에 있으므로 해당 없음.
- **성공 지표**: `share_viewed` → `fork_saved` 전환율(바이럴 루프의 핵심 지표), fork한 사람이 이후 `ticket_created`·`share_created`까지 도달하는 2차 전환(루프가 실제로 도는지).
- **엣지케이스**
  1. 이미 앱을 쓰던 기존 사용자가 남의 링크를 fork → 기존 리스트들 옆에 새 리스트로 추가될 뿐, 충돌 없음(각 리스트는 독립 id).
  2. fork한 플리를 또 fork해서 재공유(fork의 fork) → **직전 1단계만 기록·표기(확정, §3-2)** — 원작자 연쇄 추적은 MVP 미지원.
  3. 원본이 이미 운영자에 의해 takedown된 뒤 그 slug로 다시 접근 → SCREENS §7-4 "잘못된 slug" 404 상태와 동일 처리(takedown 후 레코드가 삭제되므로 자연히 같은 경로를 탄다).

## 2. 전역 내비게이션 규칙

### 2-1. 하단 TabBar

| 탭 | 라우트 | 강조 원칙 |
|---|---|---|
| 홈 | `/` | **코어(세션 플래너)가 주인공** — 기본 진입 탭이자 시각적으로 가장 강조된 탭(PRODUCT_SPEC §7-5 IA 원칙). |
| 검색 | `/search` | 보조 탭. |

- **2개 탭 고정(확정)**. "내 리스트" 탭은 두지 않는다 — 대응 라우트가 없고(`/lists` 목록 화면은 MVP 밖), 다중 리스트 전환·관리는 **홈의 리스트 스위처 BottomSheet**(SCREENS §3-3)가 담당한다. `/list/[id]`는 홈에서 진입하는 몰입형 편집 화면(TabBar 숨김)이지 탭 목적지가 아니다.
- 3번째 탭을 추가하지 않는다(발견/셀럽 탭은 후속 — BUILD_PLAN §5, PRODUCT_SPEC §7-5는 "발견은 2차 탭"이라 명시하지만 **이 MVP 범위에는 아예 없다**, 라우트조차 예약만).
- `/ticket/[id]`·`/s/[slug]`·not-found는 TabBar를 숨기는 몰입 화면(SCREENS 각 §6-1·§6-3 참조).

### 2-2. 뒤로가기 스택 규칙

- 스택 루트는 항상 **홈**이다. `/search`·`/list/[id]`는 홈에서 시작해 홈으로 돌아가는 리프 노드로 취급한다(다른 리프 노드로 직접 건너뛰지 않는다 — 예: 검색에서 뒤로가기가 리스트 상세로 가지 않고 항상 홈).
- `/ticket/[id]`도 홈에서 시작해 홈으로 돌아간다.
- `/s/[slug]`는 스택 외부 딥링크 진입점 취급 — 뒤로가기 스택이 비어 있을 수 있음을 정상으로 본다(여정⑤ 참조).
- BottomSheet·ConfirmDialog 등 오버레이는 뒤로가기(시스템 백/제스처)로 닫히는 것을 표준으로 한다(모달을 열어둔 채 실수로 앱을 벗어나는 것을 방지).
- 명시적 "닫기" 없이 뒤로가기만으로 나갈 수 없는 화면을 만들지 않는다(다크패턴 금지 목록 §3-4와 연결).

### 2-3. 딥링크 (`/s/[slug]` = 무설치 첫 진입점)

- **미설치자 여정 우선 설계**: `/s/[slug]`는 로그인·설치·가입 요구 없이 완전히 기능해야 한다(SCREENS §7이 서버 SSR만으로 완결되는 이유).
- PWA가 이미 설치된 기기에서 딥링크를 열면 설치된 앱으로 열리는지, 브라우저 탭으로 열리는지는 플랫폼별 상이 — PLATFORM_NOTES 정본(미작성) 소관, 이 문서는 "어느 쪽이든 §7 화면 요구사항은 동일해야 한다"는 원칙만 규정.
- 딥링크로 들어온 사용자에게 "앱 설치하기" 유도는 **fork 완료 후**에만 노출한다(fork 전에 설치를 강요하면 여정⑤의 핵심 가치인 "무가입 즉시 열람"이 훼손됨) — PWA 설치 배너 타이밍은 §3-3에서 상세.

### 2-4. 스크롤 복원

- 원칙: **탭 간 이동에서는 스크롤 위치를 복원하지 않는다**(각 화면 진입은 "새로 보기" 의도로 취급 — SCREENS 각 화면 §뒤로가기·스크롤 절 참조). 이는 콘텐츠가 실시간으로 바뀌는(계산·리스트) 앱 특성상 오래된 스크롤 위치가 오히려 혼란을 줄 수 있기 때문.
- 예외: 같은 화면 내에서 오버레이(BottomSheet 등)를 열었다 닫을 때는 배경 스크롤 위치를 유지한다.

## 3. 신뢰 UX

### 3-1. 공유 = 공개, 고지 시점·방식

- **원칙(API_CONTRACT §3 인용 포인터)**: "공유 = 공개다. slug를 아는 사람은 누구나 열람할 수 있고 이는 제품 의도다." 이 사실을 사용자가 **행동하기 전에** 알아야 다크패턴이 아니다.
- **고지 시점 1 — 공유 생성 전**(`/ticket/[id]`): "링크 공유"·"카톡 공유" 버튼 근처(버튼을 누르기 전 시야에 이미 들어오는 위치)에 상시 고지 문구를 둔다. 별도 확인 모달로 막지 않는다(매번 클릭을 가로막는 모달은 §3-4 다크패턴 금지 목록에 해당 — 상시 노출로 대체).
- **고지 시점 2 — 열람 시**(`/s/[slug]`): 받는 사람 입장에서도 "이 링크는 공개돼 있다"는 사실이 fork 버튼 위에 상시 노출된다(SCREENS §7-2 와이어프레임의 공개범위 고지 위치).
- 문구 원문은 `MICROCOPY §공개고지`(미작성) 몫. 이 절이 규정하는 것은 "언제·어디에 뜨는가"까지다.

### 3-2. fork 출처 표기 (확정)

- **표기 위치**: fork로 저장된 **로컬** 리스트(홈 리스트 카드·`/list/[id]` 상세)에 AttributionBadge로 출처를 남긴다. **`/s/[slug]` 공유 뷰에는 표기하지 않는다** — 공유 뷰가 곧 원본이며, 계정이 없는 구조에서 "만든 사람"을 표기할 신원 자체가 없다(COMPONENTS §16과 정합).
- **표기 내용**: 원본 공유물의 **제목** 기반("'{원본제목}'에서 가져온 플리" — 원문은 MICROCOPY §4-5 `fork.attribution`). 닉네임·작성자명은 MVP에 존재하지 않는다(PRODUCT_SPEC §7-4의 "원작자 표기"는 이 제목 기반 출처 표기로 충족하는 것으로 확정 — 신원 표기는 계정 도입 이후 과제).
- **다단계 fork(fork의 fork)**: 로컬 `forkedFrom`은 직전 원본 slug 1단계만 기록·표기한다. 원작자 연쇄 추적은 MVP 미지원(공유 payload에 출처 체인을 싣지 않음 — API_CONTRACT §4 스키마에 origin 필드 없음).

### 3-3. takedown 요청 경로

- takedown은 **운영자 몫**이며 이 문서는 사용자 여정 관점의 진입점만 규정한다: 신고/삭제 요청 경로가 앱 내 어딘가(예: `/s/[slug]` 하단 또는 별도 문의 페이지)에 있어야 한다는 요구사항만 여기서 확정하고, 구체 절차·처리 SLA·법적 근거는 `engineering/SECURITY.md`(미작성) 정본.
- 셀럽 플리 관련 takedown(PRODUCT_SPEC §7-5)은 이 MVP 범위(발견 탭 없음) 밖이라 이 문서에서 여정으로 다루지 않는다 — 일반 공유 리스트에 대한 takedown 진입점만 MVP 대상.

### 3-4. "데이터는 이 기기에만 저장됨" 정직 고지 + iOS 유실 리스크 대응

- **정직 고지**: 로그인이 없다는 것의 이면(PRODUCT_SPEC §11-4 트레이드오프)을 사용자가 알아야 한다. 어디선가 한 번은 "이 기기에서 지우거나 브라우저 데이터를 비우면 리스트가 사라질 수 있어" 계열 고지가 있어야 한다 — 시점은 **앱을 처음부터 가로막지 않는 선에서**, 예를 들어 첫 리스트 저장 직후 1회성 안내 또는 설정/정보 영역 상시 노출(어느 쪽이 나을지는 §미결).
- **iOS Safari 저장소 축출 리스크**: iOS는 일정 기간 미사용 시 IndexedDB를 브라우저가 정리할 수 있다는 알려진 제약이 있다(구체 조건·버전별 차이는 `PLATFORM_NOTES.md` 정본, 실측 필요). 여정 대응:
  1. `navigator.storage.persist()` 요청을 **적절한 시점**에 호출 — 첫 진입 즉시가 아니라, 사용자가 실제로 리스트에 곡을 담아 "지킬 가치가 있는 데이터"가 생긴 뒤(예: 곡 2~3개 담은 시점 또는 첫 리스트 저장 시)에 요청한다. 이유: 브라우저의 persist 승인 UX 자체가 사용자 관여도(engagement)에 반응하는 경우가 있고, 아무 맥락 없이 앱 진입 즉시 권한성 요청을 띄우면 거부율이 높고 신뢰도 깎임.
  2. PWA 설치 유도 배너도 같은 원칙 — **성가심 최소화 규칙**: (a) 첫 세션에는 노출하지 않는다, (b) 가치 있는 행동(예: 티켓을 처음 만들었을 때, 또는 재방문 시)을 계기로만 노출, (c) 한 번 닫으면 일정 기간(구체 일수는 실측/기획 확인 필요) 재노출하지 않는다, (d) 배너는 상시 화면 하단 요소가 아니라 dismiss 가능한 비강제 UI로 — TabBar를 가리거나 주요 CTA를 밀어내지 않는다.
  3. `pwa_install_shown`·`pwa_install_accepted` 계측으로 배너 타이밍의 실제 전환율을 데이터로 검증할 수 있게 한다(감이 아니라 지표로 성가심/효과 균형을 조정).

### 3-5. 다크패턴 금지 목록 (코덱스 리뷰 관점)

이 프로젝트가 명시적으로 하지 않는 것:

- **사전선택 동의 없음**: 공유 시 "카톡 친구에게 자동 알림" 같은 옵션을 기본 체크 상태로 두지 않는다. 모든 공유 액션은 사용자의 명시적 탭으로만 시작된다.
- **거절 어려운 모달 없음**: PWA 설치 유도·persist 요청 등은 dismiss가 명확한 버튼으로 항상 존재해야 하며, "다시 보지 않기"를 숨기거나 X 버튼을 작게/찾기 어렵게 만들지 않는다.
- **가짜 긴급성 없음**: "지금 안 하면 사라져요" 류의 카운트다운·재고 압박형 카피를 계산 결과나 공유 CTA에 쓰지 않는다.
- **취소 경로 비대칭 없음**: fork·담기처럼 "하기"가 쉬우면 "되돌리기"(리스트 삭제, fork 취소 등)도 최소 동일한 탭 수 이내여야 한다.
- **숨겨진 과금 없음**: MVP는 결제 자체가 없으므로(BUILD_PLAN §1 제외 항목) 이 항목은 현재 해당 사항 없음 — 단, 후속 구독 도입 시 이 목록에 "무료로 보이는 유료 액션 없음"을 추가해야 한다(§미결로 남김, 지금은 범위 밖).
- **오도적 공개 범위 축소 표현 없음**: "친구끼리만 볼 수 있어요" 같은, 실제로는 slug를 아는 누구나 볼 수 있는 사실과 어긋나는 카피를 쓰지 않는다(§3-1과 직결).

## 4. 오프라인 여정

| 기능 | 오프라인에서 되는가 | 근거 |
|---|---|---|
| 홈에서 현재 리스트 열람 | **됨** | Dexie 로컬(SCREENS §3-1) |
| 리스트 순서변경·삭제·노트 편집 | **됨** | 전부 로컬 쓰기(SCREENS §5-1) |
| 세션 계산(정방향·역방향, 모드 전환) | **됨** | 순수 클라 함수, 서버 호출 없음(BUILD_PLAN §3) |
| 곡 검색 | **안 됨** | Supabase RPC 필수(SCREENS §4-4) |
| 티켓 이미지(PNG) 저장 | **됨** | html-to-image 클라 렌더(SCREENS §6-4) |
| 공유 링크 생성 | **안 됨** | `shared_lists` insert가 서버 필요(SCREENS §6-4) |
| 카톡 공유 | **안 됨** | 링크 공유와 동일 스냅샷 경로에 의존 |
| fork(공유 뷰에서 저장) | **안 됨(진입 자체 불가)** | `/s/[slug]`가 SSR 필수라 오프라인이면 화면 자체를 못 연다(SCREENS §7-4) |
| 히스토리 기록·조회 | **됨** | Dexie 로컬 |

- **각 차단 지점의 안내**:
  - `/search` 진입 시 오프라인이면 검색창 대신 오프라인 안내 + "홈에서 지금 리스트로 계산은 계속할 수 있어" 계열 카피로 **할 수 있는 일로 유도**(막힌 것만 말하지 않는다).
  - `/ticket/[id]`에서는 이미지 저장 버튼은 살아있고, 링크/카톡 공유 버튼만 비활성 + 사유 안내(SCREENS §6-4) — 기능을 통째로 숨기지 않고 "지금은 안 되는 이유"를 보여줘 나중에 온라인 복귀 시 같은 자리에서 이어할 수 있게 한다.
  - `/s/[slug]`로 오프라인 상태에서 딥링크 진입 시도 시, SSR 자체가 실패하므로 브라우저 레벨의 오프라인 안내(앱이 커스텀 처리 못하는 한계)로 떨어진다는 것을 인지한다(SCREENS §7-4, PWA 서비스워커가 이 라우트까지 캐시하지 않는 한 — 오프라인 shell 캐시 범위는 ARCHITECTURE.md 정본 몫, §미결).
  - 온라인 복귀는 자동 감지(브라우저 online 이벤트)로 배너/비활성 상태를 자동 해제한다 — 사용자가 수동으로 "다시 시도"를 여러 번 누르게 하지 않는다(재시도 버튼은 있되, 자동 복귀도 병행).
