## 0. 표기 규칙 (이 문서 전체에 적용)

- **문구**: 모든 사용자 노출 문구는 이 문서에 원문으로 쓰지 않는다 → `〔문구: MICROCOPY §섹션명〕` 플레이스홀더. 실제 키는 MICROCOPY.md §4 문구 사전(화면별 표: `home.*`, `search.*`, `list.*`, `ticket.*`, `shared.*`, `common.*`, `pwa.*`)에 확정되어 있다 — 구현 시 이 화면명 가칭을 그 표의 키로 대응시킨다.
- **색**: hex 금지, 토큰 이름만(`--base`, `--surface`, `--ink`, `--ink-muted`, `--muted`, `--line`, `--point`, `--point-soft`, `--on-point`, `--gold`, `--gold-text`, `--danger` — 값·사용 규칙은 design/DESIGN_SYSTEM.md §1~§3 정본. 금액 텍스트는 항상 `--gold-text`).
- **컴포넌트**: 아래 16종에서만 골라 쓴다. 새 컴포넌트가 필요하면 이 문서에 만들지 말고 COMPONENTS.md 갱신을 먼저 요청한다(§미결에 기록).
  `Button, Chip, Stepper, SegmentedControl, PriceInput, RangeSlider, TabBar, SongRow, TicketCard, BottomSheet, Toast, Skeleton, EmptyState, ConfirmDialog, SearchInput, AttributionBadge`
- **계측 이벤트**: 아래 12종 이름만 쓴다. `app_open, song_search, song_added, calc_changed, ticket_created, share_created, share_viewed, fork_saved, history_marked, pwa_install_shown, pwa_install_accepted, return_visit`. 속성 스키마는 engineering/ANALYTICS.md 정본(미작성, §미결).
- **모션**: 이 문서의 `motion-*` 표기는 **서술용 별칭**이다. 실제 토큰 배정 정본은 DESIGN_SYSTEM §6 — 4대 핵심 인터랙션(`motion-add-spring`=담기 스프링, `motion-count-up`=숫자 카운트업, `motion-ticket-issue`=티켓 발권, `motion-page-transition`/`motion-tab-switch`=페이지·탭 전환)은 §6 배정표를 따르고, 그 외 별칭(`motion-list-*`, `motion-chip-select`, `motion-toast-in`, `motion-page-fade-in`, `motion-fork-save` 등 비핵심 모션)은 전부 `--duration-fast` + `--ease-standard` 기본값(과한 모션 금지 — BUILD_PLAN §14 "핵심 3~4곳만").
- **곡 길이**: `duration_sec`(초) 단위만 쓴다(API_CONTRACT §1-2). 화면 표기는 분:초로 파생하되 저장·계산 근거는 항상 초.
- **계산 규칙 자체**(공식·반올림·묶음요금 등)는 이 문서에 없다 → `BUILD_PLAN §6` 참조.
- **공유 payload 항목·drop 규칙**은 `API_CONTRACT §4` 참조.

## 2. 공통 규칙

- **레이아웃**: 모바일 퍼스트 단일 컬럼. 480px 이상은 중앙 정렬 `max-width: 448px` 앱 프레임, 그 바깥은 여백(배경색만). 데스크톱 전용 레이아웃은 만들지 않는다.
- **터치 타깃**: 모든 인터랙티브 요소 최소 44×44px.
- **하단 TabBar**: `/`, `/search`, `/list/[id]`(진입 시) 화면에서 상시 노출, `/ticket/[id]`·`/s/[slug]`·not-found는 몰입 화면이라 TabBar 숨김(전역 규칙은 UX_FLOWS §2).
- **robots 정책(공통 한 줄)**: 로컬 데이터 기반 화면(`/`, `/search`, `/list/[id]`, `/ticket/[id]`)은 사용자마다 내용이 달라 색인 의미가 없으므로 `noindex, nofollow` + `robots.txt`에서 크롤 자체를 막지는 않되(딥링크는 살아있어야 하므로) 메타 태그로 비색인 처리한다. `/s/[slug]`만 색인 허용.

## 3. `/` — 홈 (세션 플래너)

### 3-1. 목적·진입·데이터
- **목적 한 줄**: 지금 담은 곡으로 얼마나 걸리고 얼마 나오는지 실시간으로 보여주고, 티켓을 만들게 한다.
- **진입 경로**: 앱 최초 실행(PWA 아이콘/URL 직접), TabBar '홈' 탭, `/search`에서 담기 후 자동 복귀 없음(담기는 `/search`에 머무름, 사용자가 직접 홈으로 이동), `/ticket/[id]`·`/list/[id]`에서 뒤로가기.
- **데이터 소스**: **Dexie(로컬)만.** 현재 리스트·계산 파라미터·히스토리 전부 로컬. 서버 호출 없음(계산은 순수 클라 함수, BUILD_PLAN §6).

### 3-6. 뒤로가기·스크롤

- 홈은 내비 스택의 루트다. 홈에서 시스템 뒤로가기(브라우저/제스처)는 **앱 이탈**(PWA 종료 확인 없음, 브라우저 기본 동작) — 별도 confirm 없음(전역 규칙은 UX_FLOWS §2).
- `/search`·`/ticket/[id]`·`/list/[id]`에서 뒤로가기로 홈에 돌아오면 **스크롤 위치는 상단으로 리셋**(홈은 매번 최신 상태를 보여줘야 하는 대시보드 성격 — 스크롤 복원 대상 아님).
- 리스트 스위처 BottomSheet는 뒤로가기(모바일 시스템 백)로도 닫힌다(모달 스택 규칙, UX_FLOWS §2).

### 3-7. meta/SEO

- `title`: `싱송 — 노래방 세션 플래너`(고정, 로그인 없어 사용자별 title 분기 없음).
- `robots`: `noindex, nofollow`(§2 공통 규칙 — 로컬 데이터라 색인 무의미).
- OG 태그: 없음(공유 대상이 아님. 공유는 `/ticket`에서 만든 `/s/[slug]`가 대신함).

## 4. `/search` — 곡 검색·담기

### 4-1. 목적·진입·데이터
- **목적 한 줄**: 곡명·가수·번호·초성으로 찾아서 현재 리스트에 담는다.
- **진입 경로**: 홈의 "곡 담으러 가기" CTA, TabBar '검색' 탭. **독립 페이지**(확정 사항 — 모달로 만들지 않는다).
- **데이터 소스**: **양쪽.** 검색 조회는 Supabase RPC `search_songs`(API_CONTRACT §2-1), 담기 결과 반영은 Dexie(로컬).

### 4-2. 와이어프레임

```
┌───────────────────────────┐
│ ← 뒤로   검색              │
├───────────────────────────┤
│ [🔍 곡명·가수·번호·초성…] │  ← SearchInput
├───────────────────────────┤
│ SongRow  곡A  TJ12345 [＋] │
│ SongRow  곡B  KY 6789 [＋] │
│ SongRow  곡C  TJ 456  [✓담김]│ ← 이미 담긴 곡은 상태 다름
│ SongRow  …                 │
├───────────────────────────┤
│ 홈 | 검색                   │  ← TabBar(2탭 확정)
└───────────────────────────┘
```

### 4-3. 구성 요소 표

| 요소 | 컴포넌트 | 동작 |
|---|---|---|
| 검색창 | SearchInput | 디바운스 입력, IME 조합 중 검색 트리거 금지(PLATFORM_NOTES §IME, 미작성이나 원칙은 확정 — §미결) |
| 결과 목록 | SongRow(반복) | 우측 담기(＋) 버튼, 이미 담긴 곡은 "담김" 표시로 대체(재탭 시 제거 확인 없이 바로 토글) |
| 하단 내비 | TabBar | '검색' 강조 |

### 4-4. 상태 매트릭스

| 상태 | 무엇이 보이는가 | 복구 행동 |
|---|---|---|
| 최초(입력 전) | 검색창 포커스 안내 + 〔문구: MICROCOPY §검색-플레이스홀더〕, 결과 영역은 빈(최근 검색어 후속 기능은 MVP 범위 밖) | 타이핑 시작 |
| 로딩(디바운스 후 RPC 대기) | Skeleton(SongRow 5개 자리) | 없음(자동 해소, 200ms대 목표) |
| 빈(검색 결과 없음) | EmptyState + 〔문구: MICROCOPY §검색-결과없음〕("이 곡은 아직 없네" 계열, API_CONTRACT §5) | 검색어 수정 유도(다른 표기로 재검색) |

## 7. `/s/[slug]` — 공유 뷰(읽기전용)

| 에러(네트워크) | 이 라우트는 SSR이라 최초 접근 자체가 실패 → 브라우저 기본 오프라인 페이지 또는 Next 에러 바운더리, 앱 자체 EmptyState는 못 띄움(서버 왕복 필수 라우트 한계) | 새로고침 |
| 오프라인(화면특유) | **완전히 동작 불가**(SSR+서버 조회 필수). 이미 열람한 적 있어도 PWA 캐시 대상이 아니므로 재진입 실패 | 온라인 복귀 대기 |
| 화면특유 — 이미 이 slug를 fork한 적 있음(동일 기기) | 저장 버튼은 그대로 노출(중복 fork 차단은 MVP 범위 밖 — 매번 새 복제본 생성) | — |

### 7-5. 인터랙션 표

| 사용자 행동 | 시스템 반응 | 모션 | 계측 이벤트 |
|---|---|---|---|
| 링크 진입 | 서버 렌더 완료 표시 | `motion-page-fade-in` | `share_viewed` |
| 내 플리로 저장 탭 | 로컬 복제, `increment_fork` 호출, Toast 확인 후 홈 이동 | `motion-fork-save` + `motion-toast-in` | `fork_saved` |

### 7-6. 뒤로가기·스크롤

- 이 화면은 **딥링크 진입점**이라 뒤로가기 스택이 비어 있을 수 있다(미설치자가 카톡에서 바로 옴). 이 경우 시스템 뒤로가기는 **브라우저/카톡 인앱 이전 화면**(앱 밖)으로 나간다 — 정상 동작, 막지 않는다.
- 앱 내부에서 온 경우(자기 링크 확인)는 `/ticket/[id]`로 복귀.
- fork 성공 후에는 **홈으로 강제 이동**(뒤로가기로 다시 이 화면에 오면 "또 저장"을 반복 유도하지 않도록 UX_FLOWS §여정⑤ 참조).

### 7-7. meta/SEO

- `title`: `〔문구: 리스트제목〕 티켓 — 싱송`(동적).
- `robots`: **색인 허용**(이 화면만 유일하게 공개·불변 콘텐츠).
- **OG 필수**: `og:title`, `og:description`(곡수·1인당 요약), `og:image`(next/og로 생성한 티켓형 이미지, API_CONTRACT §2-2·BUILD_PLAN §11-4). 카톡 인앱 브라우저 미리보기가 이 화면의 실질적 첫 접점이므로 OG 이미지 200 응답이 필수 완료조건(BUILD_PLAN §11-4).

## 8. not-found — 전역 404

### 8-1. 목적·진입·데이터
- **목적 한 줄**: 존재하지 않는 경로에 온 사용자를 당황하지 않게 홈으로 되돌린다.
- **진입 경로**: 정의되지 않은 임의 URL, 삭제된 리소스에 대한 잘못된 링크(단, `/s/[slug]` 자체의 "잘못된 slug"는 §7-4에서 자체 처리하며 이 전역 404와는 별개 — 공유 뷰는 자기 화면 안에서 안내한다).
- **데이터 소스**: 없음(정적).

### 8-2. 와이어프레임

```
┌───────────────────────────┐
│ 싱송                       │
├───────────────────────────┤
│                             │
│      (일러스트: 음표 미아)  │
│  〔문구: MICROCOPY §404〕   │
│                             │
│  [홈으로 가기]  Button      │
│                             │
└───────────────────────────┘
```

### 8-3. 구성 요소 표

| 요소 | 컴포넌트 | 동작 |
|---|---|---|
| 안내 | EmptyState | 일러스트 + 카피 |
| 복귀 버튼 | Button(primary) | `/` 이동 |

### 8-4. 상태 매트릭스

| 상태 | 무엇이 보이는가 | 복구 행동 |
|---|---|---|
| 유일 상태 | 위 와이어프레임 그대로(로딩/에러 분기 없음 — 정적 페이지) | 홈으로 가기 |

### 8-5. 인터랙션 표

| 사용자 행동 | 시스템 반응 | 모션 | 계측 이벤트 |
|---|---|---|---|
| 홈으로 가기 탭 | `/` 이동 | `motion-page-transition` | — |

### 8-6. 뒤로가기·스크롤

- 시스템 뒤로가기는 이전(존재하지 않았던 URL 이전) 화면으로 정상 이동. 스크롤 해당 없음(내용이 뷰포트 안에 다 들어감).

### 8-7. meta/SEO

- `title`: `페이지를 찾을 수 없어요 — 싱송`.
- `robots`: `noindex, nofollow`.
- HTTP 상태 코드 404 유지(Next.js `not-found.tsx` 기본 동작).

| 결정 | 근거 | 기각한 대안 |
|---|---|---|
| `/search`를 독립 페이지로(모달 아님) | 프로젝트 확정 사항. 검색은 곡 카탈로그 전체를 훑는 별도 태스크 컨텍스트라 뒤로가기·URL 공유·브라우저 히스토리가 자연스러운 편이 낫다 | 모달 — 뒤로가기 스택이 꼬이고 URL로 검색 상태 공유 불가 |
| `/s/[slug]` 잘못된 slug를 전역 not-found로 안 보내고 화면 자체에서 404 처리 | 공유 뷰는 SSR 컨텍스트라 slug 유효성 판단이 그 라우트 안에서만 가능하고, 앱 브랜드(TicketCard 톤)를 유지한 안내가 전역 404보다 신뢰감을 준다 | 전역 not-found로 리다이렉트 — 맥락(어떤 링크가 잘못됐는지) 상실 |
| `/list/[id]` 존재하지 않는 id를 전역 not-found가 아닌 화면 내 EmptyState로 처리 | 이 라우트는 순수 로컬 데이터라 "서버 자원 없음"의 404 의미론과 다르다(기기 삭제·다른 기기 등) | 전역 not-found 재사용 — 로컬/서버 구분이 흐려짐 |
| 홈의 빈 상태를 온보딩으로 겸용 | PRODUCT_SPEC의 원칙(로그인·튜토리얼 없이 즉시 사용) — 별도 온보딩 화면을 만들지 않는 대신 빈 상태 카피·CTA가 그 역할을 온전히 대신해야 한다 | 별도 온보딩 슬라이드 — 첫 진입 마찰 추가, 무가입 철학과 상충 |
| 계산 결과 "큰 숫자"를 홈의 시각적 주인공으로 배치 | BUILD_PLAN §14 디자인 퀄리티 바("큰 숫자가 시각적 주인공")·PRODUCT_SPEC §5(계산기가 핵심 차별화) | 리스트를 더 크게 — 리스트는 재료, 계산이 주인공이라는 §SPEC 6 원칙과 상충 |
